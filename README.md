# 모모스 생두 계약 대시보드

Next.js + Supabase로 만든 생두 계약·물류·결제·신용장 통합 관리 시스템입니다.
빌드는 이미 검증되었습니다(`npm run build` 통과). 아래 순서대로 따라하면 실제 웹 주소를 가진
서비스로 배포할 수 있습니다. 코드를 잘 모르더라도 따라할 수 있게 단계를 최대한 풀어서 적었습니다.

## 왜 이 구조인가

- **Supabase** = 데이터베이스(Postgres) + 로그인 + 파일 저장을 한 번에 제공하는 서비스. 무료 플랜으로 시작 가능.
- **Next.js** = 화면(웹 UI)을 만드는 프레임워크. **Vercel**이라는 곳에 올리면 실제 인터넷 주소가 생긴다.
- 둘 다 무료 플랜으로 시작해서, 실제로 회사에서 쓰기 시작하면 트래픽/저장공간에 따라 유료로 전환하면 된다.

## 1단계. Supabase 프로젝트 만들기 (5분)

1. [supabase.com](https://supabase.com) 가입 → "New Project" 생성 (이름은 자유, 리전은 Northeast Asia(Seoul) 선택 권장)
2. 프로젝트가 생성되면 좌측 메뉴 **SQL Editor** 클릭
3. 이 코드 폴더의 `supabase/migrations/0001_init.sql` 파일을 열어 **전체 내용을 복사**해서 SQL Editor에 붙여넣고 **Run** 클릭
   - 이 한 번의 실행으로 테이블, 권한 규칙(RLS), 계약확정 자동화, 신용장 한도 계산 함수가 모두 만들어집니다.
4. (선택) 샘플 데이터를 보고 싶다면 `supabase/seed_optional.sql` 내용도 같은 방식으로 실행
5. 좌측 메뉴 **Project Settings → API** 에서 `Project URL`과 `anon public` 키를 복사해둔다 (3단계에서 사용)

## 2단계. 첫 계정(관리자) 만들기

1. Supabase 좌측 메뉴 **Authentication → Users → Add user** 에서 본인 이메일/비밀번호로 계정 생성
2. **Table Editor → profiles** 테이블에서 방금 만든 계정의 `role` 값을 `admin`으로 직접 수정
   (가입 시 기본값은 `buyer`이며, 이후 팀원들의 역할도 이 화면에서 admin이 직접 지정합니다 — `buyer`/`logistics`/`finance`/`admin` 중 하나)
3. 팀원 계정도 같은 방식(Add user)으로 추가하고 역할을 지정한다. 회사 내부 시스템이므로 공개 가입 화면은 만들지 않았습니다.

## 3단계. 로컬에서 실행해보기 (배포 전 확인용)

```bash
npm install
cp .env.example .env.local
# .env.local 파일을 열어 1단계에서 복사한 Project URL / anon key를 채워넣기
npm run dev
```
브라우저에서 `http://localhost:3000` 접속 → 2단계에서 만든 계정으로 로그인.

## 4단계. 실제 웹 주소로 배포하기 (Vercel)

1. 이 코드 폴더를 GitHub 저장소로 올린다 (private repo 권장 — 계약/금액 정보가 들어가는 코드이므로)
2. [vercel.com](https://vercel.com) 가입 → "Add New Project" → 방금 만든 GitHub 저장소 선택
3. **Environment Variables** 항목에 `.env.local`과 동일하게 `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY` 입력
4. Deploy 클릭 → 몇 분 후 `https://xxxx.vercel.app` 형태의 실제 주소가 생성됨
5. 팀원들에게 이 주소를 공유하면, 각자 발급받은 계정으로 로그인해서 사용 가능

이후 코드를 수정해서 GitHub에 다시 올리면 Vercel이 자동으로 재배포합니다.

## 지금 구현된 범위 (1차 MVP)

- 로그인/역할 구분 (바이어 / 물류 / 관리팀 / 관리자)
- 계약 등록 (품질, 상업조건, 결제조건 입력 + 계약서 파일 업로드)
- 계약 상태 변경 (20단계) + 메인 대시보드 칸반(6단계 그룹) + 위험 신호 카드
- **계약확정 자동화**: 상태를 "계약확정"으로 바꾸면 자동으로
  - 필수정보(계약서/금액/수량/결제조건/공급자연락처) 누락 시 에러로 막음
  - 선적 1회차 레코드 생성
  - 수입서류 9종 체크리스트 생성
  - 물류 담당자(logistics 역할 전원)에게 알림 발송, 화면 우측 상단 알림벨에 표시
- 수입서류 체크리스트 (담당자가 직접 체크)
- 커뮤니케이션 결정 기록(Decision Log)
- **역할별 필드 권한이 데이터베이스에서 실제로 강제됨**: 물류 담당자는 가격/수량/통화/결제방식을 수정 시도하면 에러,
  관리팀은 품질 정보 수정 시도하면 에러 (Airtable 등 노코드 툴에서는 이 수준의 강제가 어려웠던 부분)
- 신용장 은행별 가용한도 실시간 계산 (`lc_bank_limits` 뷰), 미래 날짜 기준 시뮬레이션 함수(`simulate_available_limit`)는
  DB에 이미 구현되어 있으나, 이걸 화면(UI)으로 보여주는 페이지는 2차 구축에서 추가 필요

## 다음 구축 단계 (2차/3차) — 스키마는 이미 준비되어 있음

데이터베이스에는 아래 항목이 이미 테이블로 존재하지만, 화면(페이지)이 아직 없습니다. 필요해지면 알려주시면
해당 화면만 추가로 만들어드릴 수 있습니다.

- `shipments` 선적 회차별 상세 입력/수정 화면, ETD/ETA 캘린더
- `payments` 결제 실행 기록 화면, 향후 30일 결제 캘린더
- `lc_banks` / `lc_ledger` 신용장 개설·상환 관리 화면, 미래 가용한도 시뮬레이션 UI
- `risks` 위험 항목 직접 등록 화면 (현재는 보기만 가능)
- 역할별 전용 대시보드(그린빈 바이어 뷰 / 김혜란 물류 뷰 / 관리팀 뷰 분리)
- 이메일/Slack 실제 알림 발송 (현재는 앱 내 알림벨까지만 구현 — Resend 등 이메일 API 연동 시 Postgres
  트리거에서 Supabase Edge Function을 호출하도록 확장 가능)

## 알아두어야 할 한계

- 빌드 시 `@supabase/ssr`가 Edge Runtime에서 일부 Node.js API를 사용한다는 경고가 뜨는데, 실제 동작에는
  영향이 없는 알려진 경고입니다.
- 현재 Storage 권한은 "로그인한 사용자는 누구나 업로드/조회 가능"으로 단순하게 설정되어 있습니다. 파일
  단위로 더 세밀한 권한이 필요해지면 버킷 정책을 추가로 다듬어야 합니다.
- 이메일 알림이 아직 없으므로, 1차 단계에서는 "앱에 로그인해서 알림벨을 확인"하는 습관이 자리잡아야
  효과가 있습니다. 팀원들이 앱을 자주 안 열면 이메일 연동을 빠르게 다음 우선순위로 올리는 게 좋습니다.
